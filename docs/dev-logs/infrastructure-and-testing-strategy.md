# 인프라 및 테스트 파이프라인 구축 아키텍처 (Dev Log)

> 본 문서는 프로젝트의 CI/CD, PR 자동화, 그리고 테스트 전략에 대한 고민과 기술적 의사결정 과정을 기록한 문서입니다. 실무 환경과 유사한 아키텍처를 구축하기 위한 방향성을 담고 있습니다.

## 1. PR 자동화 및 커밋 관리 전략

### 1.1. git push 시 PR 자동 생성에 대한 고찰
- **이슈:** `git push`를 할 때마다 자동으로 PR을 생성하도록 만들면 어떨까? 별도의 GitHub Actions Workflow가 필요한가?
- **결론:** `on: push` 조건의 별도 Workflow(`auto-pr.yml` 등)를 통해 구현은 가능합니다. 하지만 **실무에서는 권장하지 않는 안티 패턴**입니다.
- **이유:** 작업이 끝나지 않은 불완전한 상태(WIP)에서 백업 목적으로 잦은 push를 할 때마다 PR이 갱신되거나 생성되면 리뷰어에게 혼선을 줍니다.
- **이상적인 방향:** 개발자가 직접 판단하여 **Draft PR(초안)**을 생성하고, 이후의 푸시로 코드를 갱신한 뒤, 작업이 완료되었을 때 "Ready for Review"로 전환하는 수동 방식이 가장 안전하고 대중적입니다. (자동화를 하더라도 무조건 Draft PR로 열리게 설정해야 합니다.)

### 1.2. Draft PR과 커밋 히스토리 정리 (Squash)
- **이슈:** 며칠 동안 작업하며 발생한 지저분한 커밋 내역(WIP)을 Draft PR이 열려있는 상태에서 어떻게 정리할 수 있을까?
- **해결 방안:** 
  1. 로컬 브랜치에서 `git rebase -i HEAD~N` 명령어를 사용해 커밋을 깔끔하게 하나로 압축(Squash)하거나 메시지를 수정합니다.
  2. `git push --force-with-lease`를 통해 원격 브랜치에 강제 푸시합니다.
  3. GitHub은 이를 감지하여 **열려있는 Draft PR의 커밋 히스토리를 깔끔한 내역으로 자동 갱신**해 줍니다. 이후 리뷰를 요청합니다.

### 1.3. 단일 저장소(Single Repo)에서의 실무 워크플로우 시뮬레이션
- **이슈:** Upstream(공용)과 Local(개인 Fork)이 구분되지 않은 1인 개발/단일 저장소 환경에서 실무와 최대한 비슷한 워크플로우를 구성하려면?
- **해결 방안 (Feature Branch Workflow 적용):**
  - **main 브랜치 보호 (Branch Protection):** GitHub Repository 설정에서 `main` 브랜치에 대해 `Require a pull request before merging`과 `Require status checks to pass before merging(CI 성공 필수)` 옵션을 활성화하여 직접 push를 원천 차단합니다.
  - **작업 원칙:** 모든 작업은 `main`에서 파생된 `feature/*` 브랜치에서 진행하고, 스스로 PR을 열어 CI 검증 및 리뷰를 거친 뒤 **Squash and Merge**로 `main`에 병합합니다.

### 1.4. Branch Protection 관리 전략 (Rulesets vs Classic)
- **이슈:** `master`와 `develop` 두 브랜치를 동시에 보호하고, Admin 권한을 가진 생성자 본인조차도 직접 Push를 하지 못하도록 실무 환경과 동일하게 통제하려면 어떻게 해야 하는가?
- **해결 방안 및 의사결정:**
  1. **Rulesets 도입:** 기존의 'Classic branch protection rule'은 브랜치 하나당 규칙을 개별 생성해야 하는 단점이 있습니다. 유지보수 편의성과 확장성을 위해, GitHub의 최신 기능인 **Branch Rulesets**를 도입하여 하나의 규칙 묶음으로 `master`와 `develop`을 동시에 보호합니다.
  2. **Bypass list(예외 허용 목록) 제거:** 1인 프로젝트의 Admin이라 할지라도 Bypass list에 `Repository admin` 권한을 남겨두면, 터미널에서 실수로 강제 Push(`git push origin master`)를 할 경우 그대로 통과되어 버리는 치명적인 구멍(Backdoor)이 발생합니다. 완벽한 실무 CI/CD 환경 시뮬레이션을 위해 **Bypass list를 완전히 비워서**, 무조건 PR을 생성하고 CI(Playwright, Lighthouse)가 통과해야만 Merge가 가능하도록 스스로를 강제합니다.

## 2. 개발 단계별 인프라/테스트 도입 전략

### 2.1. 기능 미완성 단계에서의 파이프라인 일괄 구축 고찰
- **이슈:** 개발 초기에 Jest, GitHub Actions, CodeRabbit, SonarQube 등 파이프라인을 세팅한 김에, Playwright, Lighthouse, Google Analytics, SEO까지 모두 세팅하는 것이 좋은가?
- **결론:** 인프라의 뼈대(Pipeline)를 구축하는 것은 찬성하지만, 세부 구현은 도구의 성격에 따라 다르게 접근해야 합니다.

### 2.2. 도구별 도입 전략 및 권장 사항
- **Lighthouse CI (성능 평가) 👉 [지금 즉시 세팅 권장]**
  - **이유:** 3D 라이브러리 추가 등으로 인한 성능 저하를 조기에 발견할 수 있습니다. 완벽한 점수 달성보다는 **"성능 하한선(Baseline) 방어막"**의 용도로 CI 파이프라인에 연결하는 것이 이상적입니다.
- **Playwright (E2E 테스트) 👉 [뼈대만 세팅 후 작성 보류]**
  - **이유:** UI가 수시로 바뀌는 단계에서 세밀한 테스트 코드를 짜면, 화면 수정 시 테스트도 계속 고쳐야 하는 '유지보수 지옥(Flaky Tests)'에 빠집니다. 지금은 '메인 페이지가 렌더링되는가?' 수준의 기본 테스트만 두고, UI가 안정화(Freeze)된 이후에 구체적인 플로우를 작성해야 합니다.
- **Google Analytics 및 세부 SEO 👉 [배포 직전 세팅]**
  - **이유:** 미완성 사이트가 검색 엔진에 노출(Indexing)되는 것을 막아야 하며, 개발자의 로컬 테스트 트래픽이 GA 데이터를 오염시킵니다. 현재는 Next.js의 Metadata API 뼈대만 잡아두는 것이 좋습니다.

## 3. UI 테스트 자동화 전략 (Visual Snapshot Testing)

### 3.1. 테스트 코드 완전 자동 생성/수정의 함정
- **이슈:** UI가 변경될 때마다 Playwright 테스트 코드도 자동으로 갱신되게 만들면 유지보수 비용을 줄일 수 있지 않을까?
- **문제점 (오라클 문제):** 테스트 코드가 수정된 코드를 기반으로 자동 생성된다면, 코드에 버그가 있어도 테스트는 그 버그를 '정상'으로 간주하고 통과(Pass)해 버립니다. 테스트의 본질적인 목적(의도 검증)이 훼손됩니다.
- **문제점 (선택자 취약성):** 자동화 도구는 깨지기 쉬운 DOM 구조(예: `div > ul > li:button`)를 기준으로 코드를 짤 확률이 높아 오히려 유지보수를 방해합니다.

### 3.2. 이상적인 대안: Visual Snapshot Testing
- 개발자가 수동으로 코드를 고치는 비용을 최소화하는 가장 실무적인 방법입니다.
- **원리:** 화면의 정상적인 상태를 캡처(Golden Master)해 두고, UI가 변경될 때마다 스크립트(`--update-snapshots`)를 실행해 기준점 이미지를 일괄 업데이트합니다.
- **활용:** PR 발생 시 CI가 현재 화면과 저장된 기준점을 픽셀 단위로 비교하여, 차이가 있으면 경고를 발생시킵니다. 개발자는 로직을 짤 필요 없이 변경된 이미지가 '의도된 것인지' 눈으로만 확인(리뷰)하면 됩니다.

### 3.3. Snapshot Testing 도입 시 주요 트러블슈팅
1. **애니메이션에 의한 테스트 실패 (Flaky):** 
   - *해결:* Playwright 설정 파일(`playwright.config.ts`)에서 `animations: 'disabled'` 옵션을 켜서 스냅샷 촬영 시 렌더링을 고정시켜야 합니다.
2. **OS/브라우저 렌더링(안티앨리어싱) 오차:** 
   - *해결:* Mac(로컬)과 Linux(CI) 간의 픽셀 오차를 방지하기 위해 미세한 허용 오차율(`maxDiffPixelRatio`)을 적용하거나, 스냅샷 생성 로직을 Docker 컨테이너(Linux 환경)에서 실행합니다.
3. **저장소 용량 비대화 문제:** 
   - *해결:* 개인 프로젝트에서는 주요 페이지만 한정적으로 테스트하여 Git 용량을 관리하고, 실무/엔터프라이즈 환경에서는 Chromatic, Percy 같은 시각적 회귀 테스트 전문 SaaS 클라우드를 도입하는 것이 이상적입니다.

---

### 3.4. 해상도(Viewport) 대응 및 테스트 파편화 방지 전략
- **Playwright (시각적 검증):** 디바이스별로 세분화하는 안티 패턴을 지양하고, **데스크톱(Desktop Chrome)**과 **모바일 디자인 표준(iPhone 12, 390px)**, 그리고 **모바일 최소 방어선(iPhone 11 Pro, 375px)** 해상도만 캡처하여 검증 리소스와 ROI의 최적점을 찾습니다.
- **Lighthouse (성능 검증):** 구글의 '모바일 우선 색인(Mobile-First Indexing)' 정책에 따라 모바일 환경(기본값)을 기준으로 성능을 검사하여, 최악의 네트워크/기기 환경에서도 성능 하한선을 방어합니다.

## 4. Visual Snapshot Testing 실제 구축 가이드 (Action Plan)

> 이 섹션은 향후 `docs/plan/` 경로에 작성될 실제 구축 파이프라인의 기반이 되는 마스터 플랜입니다.

### 4.1. 진행 순서 (Step-by-Step)

**1단계: Playwright 설치 및 기본 설정 (Infrastructure 구성)**
- `apps/portfolio` 내에 `@playwright/test` 패키지 설치.
- `playwright.config.ts` 생성 및 로컬 개발 서버(`localhost:3000`) 자동 실행 설정 구성.

**2단계: 베이스라인(기준점) 스냅샷 생성 (초기 세팅)**
- 복잡한 클릭 동작 없이, 메인 페이지(`/`), Work 페이지(`/work`) 등 주요 화면으로 이동하여 `expect(page).toHaveScreenshot('page-name.png')`를 실행하는 단순한 테스트 코드 작성.
- 로컬 최초 실행 시 에러가 발생하지만, 정상적인 현재 화면을 "기준점(Golden Master) 이미지"로 자동 캡처하여 저장하도록 유도.

**3단계: 스냅샷 자동 업데이트 스크립트 추가 (핵심 아이디어 적용)**
- `package.json`에 `"test:e2e:update": "playwright test --update-snapshots"` 스크립트 추가.
- UI 변경 시 개발자는 일일이 테스트 코드를 고치는 대신, 이 명령어 한 줄로 기준점 이미지를 최신 상태로 일괄 업데이트.

**4단계: GitHub Actions CI 연동 (검증 자동화)**
- 기존 `ci.yml` 파이프라인에 Playwright 스냅샷 비교 테스트 단계를 추가.
- PR이 올라오면 CI 서버가 렌더링 화면과 저장된 기준점 이미지를 픽셀 단위로 비교하여, 오차가 있을 시 Fail 처리 및 경고 알림.

### 4.2. 추가적으로 발생하는 문제점 및 해결 방안 (Deep Dive)

**① 애니메이션/트랜지션에 의한 테스트 실패 (Flaky Tests)**
- **문제:** Framer Motion, 3D 카드 등의 애니메이션 렌더링 프레임 오차로 인해 매번 테스트가 실패하게 됨.
- **해결 방안:** Playwright 설정 파일(`playwright.config.ts`)에 **`animations: 'disabled'`** 옵션을 전역으로 설정하여, 테스트 실행 시 모든 애니메이션을 강제로 정지시킨 상태에서 스냅샷을 촬영.

**② OS 및 브라우저 렌더링 차이에 따른 픽셀 오차**
- **문제:** Mac(개발자 로컬)과 Linux(CI 서버) 환경 간의 폰트 안티앨리어싱 및 브라우저 렌더링 엔진 미세 오차로 인한 실패.
- **해결 방안:** 허용 오차율(`maxDiffPixelRatio: 0.05` 등)을 적용하여 미세한 차이를 무시하거나, 스냅샷 갱신 및 검증 자체를 **Linux 기반의 Docker 컨테이너 내부에서 일관되게 실행**하도록 아키텍처를 구성.

**③ Git 저장소 용량 비대화 문제**
- **문제:** PNG 이미지 파일이 Git 히스토리에 누적되면서 저장소 용량이 기하급수적으로 커짐.
- **해결 방안:** 개인 포트폴리오 수준에서는 핵심 페이지만 테스트하여 용량을 관리. 실무 엔터프라이즈 환경으로 스케일업 시에는 Git에 이미지를 저장하지 않고 Chromatic, Percy, Argos와 같은 시각적 회귀 테스트(Visual Regression) 전용 클라우드 SaaS를 도입.

### 4.3. 최종 요약 및 종합 제언

현재 시점에서 Playwright의 세부적인 E2E 플로우 테스트 작성은 개발 속도를 저하시키므로 **지양**하되, **애니메이션 비활성화 옵션을 포함한 스냅샷 기준점 캡처 파이프라인(1~4단계)**을 최우선으로 구축해야 합니다. 

이를 통해 코드를 직접 수정하지 않고도 UI 회귀(Regression) 버그를 완벽하게 방어할 수 있으며, 이 구조 자체가 면접관에게 **"테스트 유지보수 비용과 개발 생산성 간의 최적의 밸런스를 아는 엔지니어"**임을 증명하는 강력한 무기가 될 것입니다. 향후에는 `Husky`를 통한 커밋 전 단위 테스트 실행과 `Turborepo`의 CI 캐싱 극대화를 연계하여 완벽한 모노레포 자동화 생태계를 완성하는 것을 추천합니다.
