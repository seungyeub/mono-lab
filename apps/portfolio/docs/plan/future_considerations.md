## 1. 데스크톱 환경 전체 폰트 스케일업(Scale-up) 고려

**배경 (Context):**

- FAQ 섹션 리디자인 과정에서 이미지를 제거하고 1단 Full-width 구조(Route B)를 적용하는 실험을 진행함.
- 1단 구조 특성상 1440px 이상의 넓은 데스크톱 화면에서는 기존의 작은 텍스트(`text-base` 수준)가 너무 넓게 퍼져 가독성이 떨어지고 휑해 보이는 현상이 발생함.
- 이를 해결하기 위해 FAQ 아코디언의 텍스트 크기를 거대하게(`text-3xl` 등) 키워 타이포그래피 포스터처럼 연출하면 시각적 문제는 해결되나, **사이트 내 다른 섹션(Works, About 등)과의 타이포그래피 위계질서 및 일관성이 심각하게 훼손**된다는 딜레마에 부딪힘.

**고민 사항 (Dilemma):**

- 특정 섹션(FAQ)의 시각적 완성도를 위해 해당 섹션만 글씨를 뻥튀기하는 것은 근본적인 해결책이 아님. (디자인 시스템 붕괴)
- 만약 FAQ와 같은 1단 브루탈리즘/미니멀리즘 스타일을 데스크톱에서 유지하고 싶다면, **데스크톱 화면 전체(`lg` 브레이크포인트 이상)의 기본 Body 폰트 크기와 Heading 스케일을 전체적으로 일괄 상향 조정(Scale-up)하는 대대적인 작업이 필요**함.

**향후 작업 방향 (Next Steps):**

- [ ] 현재의 오리지널 폰트 사이즈를 유지하며 '2단 오프셋'이나 'Max-width 래퍼'를 사용할 것인지, 아니면 데스크톱 환경 전체를 브루탈리즘 스케일로 키울 것인지 최종 결정하기.
- [ ] 만약 전체 스케일업을 결정할 경우, `WorksSection`, `HeroSection` 등의 텍스트 크기를 모두 일괄 조정할 수 있도록 Tailwind의 글로벌 타이포그래피 토큰(예: `:root`의 폰트 스케일 변수) 또는 컴포넌트별 반응형 클래스 재정의 작업 진행하기.

## 2. 3D 인터랙션 UX 마이크로카피(Microcopy) 개선 (DRAG → GRAB/PULL)

**배경 (Context):**

- 3D 사원증 캔버스(`InteractiveCardCanvas`) 영역에 마우스를 올릴 때 표시되는 커스텀 커서 텍스트가 현재 `DRAG`로 설정되어 있음.
- `DRAG`는 웹 표준 관점에서는 직관적이나, 물리 엔진(Rapier)이 적용되어 묵직하게 매달려 흔들리는 3D 물리 객체라는 감성적 맹점에서는 다소 건조하고 기술적인 인상을 줌.

**고민 사항 (Dilemma):**

- 단순한 UI 컴포넌트가 아니라 '만질 수 있는 사물'이라는 몰입감을 유저에게 주려면, 좀 더 물리적이고 장력(Tension)이 느껴지는 동사가 필요함.
- `PULL`(당기다): 매달린 물체의 중력과 장력을 극대화하여 피드백을 기대하게 만듦.
- `GRAB`(잡다): 3D/VR 환경에서 가장 직관적인 물리적 단어로 무의식적인 행동을 강하게 유도함.

**향후 작업 방향 (Next Steps):**

- [x] 추후 디자인 폴리싱(Polishing) 단계에서 사원증 커스텀 커서 텍스트를 `DRAG`에서 `GRAB` 또는 `PULL`로 교체하여 UX 테스트 진행.
- [x] 단어 변경이 결정되면, 커스텀 커서의 원형 크기 및 폰트 사이즈 등의 레이아웃이 4글자 길이에 맞게 어색함 없이 유지되는지 확인 후 최종 반영(`useCursorStore`의 상태 값 또는 해당 영역 hover 로직 수정).

## 4. Skills Section 시각적 구분선(White Line) 제거 검토

**배경 (Context):**

- 현재 Skills Section 내부의 각 영역이나 요소들을 구분하기 위해 얇은 흰색 선(White Line)이 시각적 테두리(또는 디바이더) 역할로 사용되고 있음.
- 프로젝트 전체의 미니멀리즘과 공간감을 더 강조하고, 불필요한 시각적 노이즈를 줄이기 위해 해당 선을 제거하는 방안을 고려 중임.

**고민 사항 (Dilemma):**

- 선을 완전히 제거할 경우, 자칫하면 항목 간의 시각적 경계가 너무 모호해져서 정보의 그룹화(가독성)가 떨어질 위험성이 존재함.
- 명시적인 선(Line)에 의존하지 않더라도 여백(Spacing), 타이포그래피의 크기 대비, 미세한 배경 명도 차이 등을 통해 경계를 충분히 인지시킬 수 있는 레이아웃 밸런싱이 요구됨.

**향후 작업 방향 (Next Steps):**

- [ ] 브라우저 개발자 도구 등을 활용하여 임시로 흰색 선(border-white/20 등)을 제거해 본 후, 전체적인 디자인 밸런스와 시각적 흐름(Flow) 평가하기.
- [ ] 선 제거가 긍정적으로 판단될 경우, 선을 대체하여 영역을 구분해 줄 수 있도록 마진(Margin)/패딩(Padding) 규격을 재조정하고 코드에 반영하기.

- 프로젝트 전체의 트렌디하고 역동적인 분위기를 이어가면서도, 정보를 입력하고 전송하는 'Contact' 본연의 사용성(UX)을 해치지 않는 균형점을 찾아야 함.

**향후 작업 방향 (Next Steps):**

- 현재 Header 메뉴와 서브 페이지 라우팅에 `Gallery`가 포함되어 있으나, 포트폴리오의 목적(구직, 프리랜싱, 클라이언트 어필 등)을 고려했을 때 `Gallery`보다는 전문적인 이력과 성과를 보여주는 `Resume`(이력서) 페이지가 더 적합할 수 있음.

**고민 사항 (Dilemma):**

- `Resume` 페이지를 추가할 경우, 기존 Notion이나 PDF 형식의 이력서 링크로 외부 라우팅을 시킬 것인지, 아니면 포트폴리오의 일관된 디자인 시스템(Glassmorphism 등)이 적용된 전용 웹 이력서 페이지를 앱 내부 라우트로 직접 구현할 것인지 결정이 필요함.
- 기존 `Gallery` 페이지가 담당하던 시각적 포트폴리오 성격을 버리는 대신, 정량적이고 텍스트 위주인 `Resume`가 타겟 유저(채용 담당자)에게 실질적인 효용을 줄 수 있을지 목적성에 대한 밸런스 평가가 요구됨.

**향후 작업 방향 (Next Steps):**

- [ ] 포트폴리오의 타겟 오디언스 니즈를 고려하여 `Gallery` 유지 vs `Resume` 신설에 대한 최종 방향성 결정.
- [ ] `Resume` 페이지 도입 확정 시, 내부 라우팅(`/resume`) 페이지 직접 개발 vs 외부 링크(Notion/PDF) 연결 방식 확정.
- [ ] Header 컴포넌트 내부 네비게이션 메뉴(`Nav Links`) 항목 교체 및 `next/link` 속성 업데이트.
