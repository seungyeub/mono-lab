# PRD: 3D Interactive Hero Section Card

## 1. 개요 (Overview)
본 문서는 `apps/portfolio` 프로젝트의 메인 진입점인 `HeroSection`의 우측 이미지 슬롯을 기존 정적 이미지에서 인터랙티브한 3D 물리 기반 사원증(ID Card) 모델로 교체하기 위한 제품 요구사항 및 구현 전략을 정의합니다. 참고 프로젝트(`fattahmaulana/3D_CARD`)의 물리 엔진과 3D 렌더링 방식을 차용하여 사용자에게 강렬하고 동적인 첫인상을 제공하는 것을 목표로 합니다.

## 2. 화면 및 UX 설명 (Visual & UX Details)
- **레이아웃 배치**: 
  - 화면 좌측: 기존과 동일하게 소개 텍스트, 타이포그래피, CTA(Call to Action) 버튼들이 위치합니다.
  - 화면 우측 (이미지 슬롯): 단조로운 2D 이미지가 아닌, 3D 공간(Canvas)이 렌더링됩니다.
- **3D 시각 요소**:
  - 화면 상단에 고정된 랜야드(목걸이 줄)에 매달려 있는 형태의 3D 사원증/명찰이 렌더링됩니다.
  - 마치 실제 스튜디오 조명을 받은 것처럼 플라스틱, 메탈 클립, 랜야드 천 소재의 질감이 사실적으로 표현됩니다 (빛 반사, 그림자 등).
- **사용자 상호작용 (Interaction)**:
  - **마우스 호버**: 카드 위로 마우스를 올리면 커서가 'grab'(움켜쥐기) 아이콘으로 변경되며 상호작용 가능함을 암시합니다.
  - **드래그 앤 드롭**: 카드를 클릭한 상태로 마우스를 움직이면(드래그) 실제 물리 법칙이 적용되어 줄이 당겨지며 카드가 마우스를 따라옵니다.
  - **물리 시뮬레이션**: 드래그를 놓으면 중력과 줄의 탄성에 의해 카드가 자연스럽게 흔들리며 제자리로 돌아갑니다. 스크롤이나 마우스의 미세한 움직임에도 카드가 반응하여 동적인 느낌을 줍니다.

## 3. 구현 전략 및 아키텍처 (Implementation Strategy)
**주의: 본 PRD는 구현 방향성을 제시하며, 직접적인 코드는 포함하지 않습니다.**

### 3.1 필요 기술 스택 (Dependencies)
- **Three.js**: WebGL 기반 3D 렌더링 코어 라이브러리
- **@react-three/fiber**: React 환경에서 Three.js를 선언적으로 다루기 위한 렌더러
- **@react-three/drei**: 3D 모델 로딩, 조명, 환경 설정 등을 돕는 유틸리티 라이브러리
- **@react-three/rapier**: React Three Fiber를 위한 고성능 3D 물리 엔진 라이브러리 (중력, 충돌, 조인트 연결 처리)
- **meshline**: 랜야드(목걸이 줄)와 같이 두께가 있는 선을 부드럽게 렌더링하기 위한 확장 라이브러리

### 3.2 필요 에셋 (Assets)
- **3D 모델 (`.glb` 또는 `.gltf`)**: 카드 보드, 메탈 클립, 연결 부품이 포함된 최적화된 3D 파일
- **텍스처 이미지**: 랜야드 줄에 입힐 패턴/로고 이미지 및 카드 앞면에 들어갈 포트폴리오 프로필 텍스처

### 3.3 컴포넌트 구조 (Component Architecture)
1. **HeroSection 컴포넌트 수정**:
   - 우측 이미지 영역을 감싸는 컨테이너를 `relative`, `w-full`, `h-full` 상태로 구성합니다.
   - **지연 로딩 (Lazy Loading)**: 
     - 3D 렌더링 라이브러리(Three.js, 물리 엔진 등)는 무겁고 브라우저 API(Window)에 의존하기 때문에 서버 사이드 렌더링(SSR) 환경에서 에러를 유발할 수 있습니다.
     - 이를 해결하기 위해 `next/dynamic` 함수를 사용할 때 `{ ssr: false }` 옵션을 주어, 클라이언트 브라우저에 도달한 이후에만 3D 컴포넌트가 비동기적으로 로드되고 렌더링되도록 처리합니다.
   - **등장 애니메이션 (Entry Animation)**:
     - 3D 모델이 로딩 완료되어 렌더링되는 시점에, 사원증이 공중에서 위에서 아래로 자유 낙하하듯 등장하는 물리적 연출을 더합니다.
     - 사원증 컴포넌트의 초기 생성 좌표(RigidBody의 Y축 위치)를 화면 밖 위쪽 공간으로 설정해두면, 로드가 끝남과 동시에 물리 엔진의 중력을 받아 자연스럽게 아래로 떨어집니다.
     - 랜야드(로프 조인트)가 허용하는 최대 길이에 도달하는 순간, 로프 장력에 의해 툭 걸리며 특정 위치에서 살짝 바운스(반동)되면서 최종 위치에 안정적으로 안착하는 시각적 쾌감을 제공합니다.
2. **InteractiveCardCanvas 컴포넌트 (신규)**:
   - `@react-three/fiber`의 `<Canvas>` 컴포넌트를 렌더링합니다.
   - `<Environment>`, `<ambientLight>`, `<Lightformer>` 등을 배치하여 부드러운 배경과 금속 재질의 사실적인 반사광을 설정합니다.
   - `<Physics>` 태그로 물리 엔진 공간을 생성하고, 그 내부에 실제 카드 요소인 `<Band>` 컴포넌트를 렌더링합니다.
3. **Band (물리적 연결체) 컴포넌트 (신규)**:
   - **물리 강체 (RigidBody)**: 랜야드의 시작점(화면 상단 고정), 중간 조인트들, 그리고 카드 본체를 각각 독립된 RigidBody로 선언합니다.
   - **조인트 (Joints)**: `useRopeJoint`(로프 관절)와 `useSphericalJoint`(구체 관절)를 사용하여 강체들을 연결합니다. 이를 통해 자연스럽게 흔들리는 목걸이 형태를 구현합니다.
   - **인터랙션 바인딩**: 3D 카드 모델 오브젝트에 `onPointerDown`, `onPointerUp`, `onPointerMove` 이벤트를 할당합니다. 드래그 이벤트 발생 시 중력을 무시하고 마우스 좌표(Kinematic Position)를 따라가도록 상태를 업데이트하여 잡아당기는 효과를 줍니다.
   - **MeshLine 시각화**: 매 프레임(`useFrame`)마다 보이지 않는 물리 조인트들의 위치를 추적하여 부드러운 곡선(CatmullRomCurve3)을 계산하고, 이를 기반으로 랜야드 끈(MeshLine)을 시각적으로 렌더링하여 강체들을 이어줍니다.

### 3.4 성능 최적화 및 주의사항 (Performance & Optimization)
- **에셋 압축**: 3D 모델(.glb)은 Draco 압축 등을 통해 파일 용량을 최소화하여 초기 로딩 속도 저하를 방지합니다.
- **동적 로딩 (SSR 우회)**: 3D Canvas 및 브라우저 API에 의존하는 물리 엔진 코드는 서버 사이드 렌더링(SSR) 시 에러를 유발하므로, 반드시 클라이언트 측 컴포넌트로만 작동하게 처리해야 합니다 (`'use client'` 및 동적 임포트).
- **반응형 디자인 (Responsive)**: 모바일 등 작은 화면에서는 3D 오브젝트가 화면을 가리거나 화면 밖으로 크게 벗어나지 않도록, `Canvas`의 Camera FOV(시야각)를 조절하거나 3D Group의 스케일을 뷰포트 크기에 맞게 동적으로 스케일링하는 로직을 추가해야 합니다.

## 4. 코드 작성 가이드라인 및 주석 정책 (Code Convention)
본 3D 컴포넌트는 물리 엔진과 Three.js 렌더링 로직이 결합되어 있어 코드가 매우 복잡해질 수 있습니다. 추후 유지보수 및 협업을 위해 다음 원칙을 준수하여 코드를 작성합니다.
- **상세한 주석 작성 (Mandatory Comments)**: 
  - 물리 강체(RigidBody)의 생성 목적, 각 조인트(RopeJoint, SphericalJoint)가 연결되는 원리와 초기 좌표 설정에 대해 상세한 설명을 주석으로 남깁니다.
  - 마우스 이벤트(`onPointerDown` 등)에서 Kinematic Position을 계산하여 2D 화면 좌표를 3D 공간 좌표로 변환하는 수학적 로직의 원리와 수식을 상세히 기록합니다.
  - MeshLine을 이용해 동적으로 곡선을 그리는(`CatmullRomCurve3`) 계산 파트에도 어떤 물리 조인트 위치를 참조하는지 주석으로 명시합니다.
- **상수화 (Constants)**: 마찰력, 중력 값, 로프 길이 등의 물리 파라미터는 매직 넘버로 방치하지 않고 상단에 변수로 빼내어 명시적인 주석을 추가합니다.

## 5. 보완 필요 사항 및 예외 처리 (Missing & Recommended Features)
현재 기획에서 추가로 고려하면 제품의 완성도를 크게 높일 수 있는 보완 항목들입니다.
- **의도된 여백 (로딩 스피너 생략)**: 3D 모델(.glb)과 물리 엔진 로딩은 네트워크 및 기기 성능에 따라 0.5초에서 수 초가 걸릴 수 있습니다. 하지만 **'화면 위에서 사원증이 떨어지는 연출'**을 기획했기 때문에, 로딩 중에 스켈레톤 UI나 스피너를 보여주면 카드가 위에서 떨어지는 극적 연출의 몰입감이 깨질 수 있습니다. 따라서 로딩 중에는 해당 영역을 **자연스러운 빈 공간으로 유지**하고(레이아웃 붕괴 방지용 크기만 확보), 로딩 완료 시 카드가 깜짝 등장하며 떨어지게 하여 시각적 즐거움을 극대화합니다.
- **폴백 UI (Fallback UI)**: WebGL을 지원하지 않는 브라우저나 낮은 사양의 기기에서 3D 렌더링 중 크래시가 날 경우 화면 전체가 멈추는 것을 방지해야 합니다. Error Boundary를 적용하여 실패 시 기존 2D 프로필 이미지를 띄워주는(Fallback) 대비책이 필요합니다.
- **웹 접근성 (Accessibility / a11y)**: 3D `<Canvas>` 내부는 스크린 리더가 해석하지 못합니다. 시각 장애인이나 마우스 사용이 어려운 사용자를 위해 Canvas 컨테이너에 `aria-label`이나 숨겨진 `<span>` 텍스트로 '인터랙티브 3D 사원증 뷰어'라는 대체 설명을 제공해야 합니다.
